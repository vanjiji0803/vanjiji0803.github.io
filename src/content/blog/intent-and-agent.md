在讨论 AI agent 的时候，大家很容易直接跳到工具调用、规划、记忆、RAG、工作流编排这些工程细节。但我越来越觉得，agent 系统里最容易被低估的一层，其实是 intent，也就是“意图”。

意图不是一句 prompt 的字面意思。用户说“帮我整理一下这篇论文”，表面上是整理，真正的意图可能是：快速判断这篇论文值不值得读；提取可以放进组会汇报的技术点；或者找出方法部分有没有可复现的漏洞。句子相同，意图不同，agent 应该采取的行动就完全不同。

所以我更愿意把 intent 看成用户目标和系统行动之间的一份隐含契约。它回答三个问题：

1. 用户真正想完成什么？
2. 哪些约束不能被破坏？
3. 当执行过程遇到不确定性时，系统应该优先保护什么？

这和传统的“分类意图”有一点关系，但不是同一件事。早期对话系统里的 intent classification，通常是把用户输入分到固定标签里，比如查询天气、预订机票、取消订单。这个做法在封闭任务里很有用，但对于 agent 来说还不够。因为 agent 不只是选择一个回复模板，而是要分解任务、调用工具、读写文件、访问外部 API，甚至跨多个步骤修正自己的计划。

这时 intent 就变成了执行过程中的导航信号。

比如用户说：“把这个项目部署一下。” 一个弱 agent 可能马上运行部署命令。一个更好的 agent 会先理解意图边界：用户是想本地预览，还是发布到生产？是否允许修改配置？是否需要先跑测试？有没有未提交的本地改动？这些问题不是形式主义，而是在保护用户真正的目标。部署不是目的，可靠地把正确版本上线才是目的。

在 agent 架构里，intent 通常会影响四个层面。

第一是任务分解。明确 intent 之后，agent 才知道哪些步骤必要，哪些步骤只是看起来勤快。用户想“理解失败原因”，就应该先查日志和配置；用户想“马上修好并发布”，就需要进入修改、验证、提交、部署链路。

第二是工具选择。相同输入在不同 intent 下会调用完全不同的工具。要解释概念，可能只需要阅读上下文和组织语言；要发布博客，就需要编辑文件、构建、提交、推送。

第三是风险控制。intent 决定哪些动作需要确认。比如删除文件、覆盖远端、自动发布内容，风险都不一样。一个可靠的 agent 不能只问“我能不能做”，还要问“这是否仍然符合用户最初的意图”。

第四是错误恢复。真实系统一定会失败：API key 缺失、构建报错、权限不足、网络超时。没有 intent 的 agent 只会报错；有 intent 的 agent 会判断下一步怎么补救。如果目标是发布文章，构建失败时就应该定位构建问题，而不是停在“命令失败”四个字上。

我认为 agent 的核心不是“会调用很多工具”，而是能在多步执行中持续保持意图不漂移。工具调用让 agent 有手，规划让 agent 有路线，而 intent 让 agent 知道自己为什么在走这条路。

这也是为什么好的 agent 体验有时候不像一个更聪明的聊天机器人，而像一个靠谱的协作者：它不只是听见你说了什么，还能记住你真正要完成什么，并在细节开始变复杂的时候，仍然替你守住那个目标。
