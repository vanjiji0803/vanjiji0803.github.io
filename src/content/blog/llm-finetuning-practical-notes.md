很多团队第一次考虑大模型微调时，动机都很相似：模型不知道我们的业务知识，所以把内部文档拿去 fine-tune 一下。这个直觉可以理解，但通常不是一个好的起点。微调当然会改变模型参数，但它不是一个稳定、可查询、可更新的知识库。把大量事实性知识压进权重里，代价高、可控性弱，而且一旦文档变了，就很难追踪模型到底记住了什么、忘掉了什么。

我更倾向于把微调看成一种“行为塑形”工具，而不是“知识注入”工具。它适合解决的问题包括：固定输出格式、学习特定任务的推理步骤、适应领域语言风格、降低 prompt 复杂度、让模型更稳定地调用某类工具，或者让小模型在窄任务上接近大模型的表现。如果目标是让模型回答最新制度、产品参数、病例资料或技术文档，RAG、数据库和工具调用通常应该优先。

一个比较清晰的判断标准是：如果答案来自一批会持续变化的外部资料，优先做检索；如果答案来自一种稳定的处理方式，才考虑微调。

比如“根据这份设备日志判断是否需要重启服务”更像行为模式，可以微调；“回答某台设备当前固件版本是多少”更像事实查询，应该查数据库。前者需要模型学会读日志、识别异常模式、给出操作建议；后者需要模型拿到准确数据。

工程上最常见的路线是 SFT，也就是 supervised fine-tuning。准备一批输入输出样本，让模型学习在特定上下文下应该怎么回答。真正困难的地方往往不是训练命令，而是数据。低质量样本会把模型带歪：输出啰嗦、格式不一致、错误推理被当成标准答案、边界情况缺失。一个 1,000 条高质量、结构一致、覆盖典型失败模式的数据集，通常比 50,000 条自动拼出来的脏数据更有价值。

LoRA 和 QLoRA 的意义在于降低微调成本。LoRA 不直接更新全部参数，而是在部分线性层旁边加低秩适配矩阵，只训练这小部分参数。直觉上，它不是重写整张地图，而是在关键路径上加一些可学习的修正。QLoRA 再进一步，把基座模型量化到低比特，用更少显存完成训练。它们很适合个人或小团队做领域适配，但也带来一个现实问题：适配器越多，版本管理、合并策略和线上路由就越复杂。

微调前我通常会先问四个问题。

第一，能不能用 prompt 解决？如果只是少量格式控制，system prompt、few-shot examples、结构化输出 schema 可能已经够了。

第二，能不能用 RAG 解决？如果问题依赖外部知识，检索增强比微调更可更新、更可解释。

第三，是否有稳定评测集？没有 eval 就开始微调，本质上是在凭感觉调模型。至少要有一组固定测试样本，覆盖正常输入、边界输入、脏输入和高风险输入。

第四，线上失败怎么回滚？微调模型不是 notebook 里的实验产物，它上线后需要版本号、训练数据快照、评测报告、灰度策略和回滚路径。

评测也不能只看“回答像不像”。对于技术任务，我更关注几类指标：格式是否稳定，关键字段是否缺失，工具调用参数是否正确，引用依据是否真实，拒答边界是否合理，长上下文下是否丢失约束。对于 agent 场景，还要看 trace：模型是否选择了正确工具，失败后是否恢复，是否在多步执行中保持用户 intent 不漂移。

微调也可能让模型变差。最常见的是过拟合到模板，导致模型遇到稍微不同的问题就僵硬；或者过度学习训练集里的口吻，让回答看似专业但判断力下降。还有一种隐蔽问题是能力遗忘：窄任务表现上升，但通用推理、拒答、安全边界或多语言能力下降。所以每次微调后都应该跑一组“保持能力”的回归评测，而不是只测目标任务。

我的经验是，微调最适合放在一个更大的系统里，而不是单独承担全部智能。RAG 负责知识更新，工具负责确定性操作，规则和 schema 负责边界约束，微调负责让模型更稳定地遵循任务习惯。这样系统的每一层都有明确责任，也更容易定位问题。

如果一句话总结：不要为了让模型“知道更多”而微调，要为了让模型“做得更稳定”而微调。知识应该尽量外置，行为可以适度内化。这个区分看起来简单，但会直接决定一个 LLM 系统是可维护的工程，还是一团难以解释的参数魔法。
